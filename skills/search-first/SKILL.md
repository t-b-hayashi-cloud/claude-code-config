# コーディング前に検索する

新機能の実装前に既存のソリューションを検索するワークフローを体系化します。

## 使用タイミング

- 既存のソリューションがありそうな新機能を始めるとき
- 依存関係や統合を追加するとき
- ユーザーが「X機能を追加して」と言ってコードを書こうとするとき
- 新しいユーティリティ、ヘルパー、抽象化を作成する前

## ワークフロー

```
┌─────────────────────────────────────────────┐
│  1. ニーズ分析                               │
│     必要な機能を定義する                      │
│     言語・フレームワーク制約を特定する         │
├─────────────────────────────────────────────┤
│  2. 並列検索                                 │
│     ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│     │  PyPI /  │ │  MCP /   │ │ GitHub / │  │
│     │  conda   │ │  Skills  │ │  Web     │  │
│     └──────────┘ └──────────┘ └──────────┘  │
├─────────────────────────────────────────────┤
│  3. 評価                                     │
│     候補をスコアリング（機能、保守性、         │
│     コミュニティ、ドキュメント、ライセンス）   │
├─────────────────────────────────────────────┤
│  4. 判断                                     │
│     ┌─────────┐  ┌──────────┐  ┌─────────┐  │
│     │ 採用     │  │  拡張/   │  │ カスタム │  │
│     │そのまま  │  │  ラップ  │  │  実装   │  │
│     └─────────┘  └──────────┘  └─────────┘  │
├─────────────────────────────────────────────┤
│  5. 実装                                     │
│     パッケージインストール / MCP設定 /        │
│     最小限のカスタムコードを書く              │
└─────────────────────────────────────────────┘
```

## 判断マトリックス

| シグナル | アクション |
|--------|--------|
| 完全一致、良好なメンテナンス、MIT/Apache | **採用** — そのままインストールして使用 |
| 部分一致、良い基盤 | **拡張** — インストール + 薄いラッパーを書く |
| 複数の弱い一致 | **組み合わせ** — 2〜3の小さなパッケージを組み合わせる |
| 適切なものが見つからない | **実装** — カスタム実装（ただし調査に基づく） |

## 使い方

### クイックモード（インライン）

ユーティリティを書いたり機能を追加する前に、以下を確認：

0. すでにリポジトリに存在するか？ → 関連モジュール/テストを `grep` で検索
1. よくある問題か？ → PyPI/conda で検索
2. MCPがあるか？ → `~/.claude/settings.json` を確認して検索
3. スキルがあるか？ → `~/.claude/skills/` を確認
4. GitHubに実装/テンプレートがあるか？ → メンテナンスされているOSSを検索してから新規コードを書く

### フルモード（エージェント）

非trivialな機能には、調査エージェントを起動：

```
Task(subagent_type="general-purpose", prompt="
  以下のためのツールを調査する: [説明]
  言語/フレームワーク: Python, pandas, BigQuery
  制約: [任意]

  検索先: PyPI, conda, MCPサーバー, Claude Codeスキル, GitHub
  返却: 推薦付きの比較表
")
```

## plannerエージェントとの統合

### plannerエージェントと一緒に
プランナーは実装フェーズ前に調査を実施する：
- 利用可能なツールを特定
- 実装計画に組み込む
- 「車輪の再発明」を避ける

### architectエージェントと一緒に
アーキテクトは以下のために調査を参照：
- 技術スタックの決定
- 統合パターンの発見
- 既存の参照アーキテクチャ

## 具体例（Python データ分析向け）

### 例1: 「CSVデータのバリデーションを追加したい」
```
ニーズ: pandasのDataFrameをスキーマで検証する
検索: PyPI "dataframe validation schema"
発見: pandera (スコア: 9/10), great_expectations (スコア: 8/10)
判断: ADOPT — uv add pandera
結果: カスタムコードゼロ、実績のあるソリューション
```

### 例2: 「BigQueryクエリのコストを試算したい」
```
ニーズ: クエリ実行前にスキャン量を推定する
検索: google-cloud-bigquery Python docs "dry run"
発見: client.query(..., job_config=QueryJobConfig(dry_run=True))
判断: ADOPT — 公式SDKにすでにある
結果: 追加パッケージ不要
```

### 例3: 「モデルの実験管理をしたい」
```
ニーズ: LightGBMの実験とメトリクスを追跡する
検索: PyPI "experiment tracking mlflow optuna"
発見: MLflow (重い), optuna-dashboard, wandb
判断: EXTEND — optuna + pandasでシンプルに記録
結果: 既存のoptunaを活用
```

## アンチパターン

- **すぐにコードを書く**: 存在するか確認せずにユーティリティを書く
- **MCPを無視する**: MCPサーバーが機能を提供しているか確認しない
- **過度なカスタマイズ**: ライブラリをラップしすぎてメリットを失う
- **依存関係の膨張**: 1つの小機能のために巨大なパッケージをインストール

## このプロジェクト固有の注意

- `uv add {package}` でパッケージを追加（pip installは使わない）
- BigQuery関連: `google-cloud-bigquery` が公式、他のBQラッパーは原則不要
- ML系: `lightgbm`, `scikit-learn`, `optuna` が利用可能か `pyproject.toml` で確認してから追加
- `pyproject.toml` の `[project.dependencies]` を確認して重複を避ける
